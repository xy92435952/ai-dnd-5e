## 1. 产品概述

### 1.1 产品定位

**AI跑团平台**是一个基于 DnD 5e 规则、运行于浏览器的单人跑团游戏。用户上传模组文件后，AI 担任地下城主（DM）、并扮演平衡配置的AI队友，让玩家无需凑人、无需真人DM，即可独立完成完整的桌游跑团体验。

### 1.2 核心价值主张

| 痛点 | 本产品的解法 |
|------|------------|
| TRPG最难的事是凑人 | AI DM + AI队友，一人即可开团 |
| AI叙事平台没有规则骨架，内容飘散 | 完整实现 DnD 5e 规则引擎（骰子/检定/战斗） |
| 战斗无可视化，体验差 | 网格战斗地图，实时显示位置与状态 |
| 好的剧本难找，不能自定义 | 模组上传系统，支持PDF/DOCX/MD/TXT |
| 现有中文跑团平台无AI功能 | 中文优先设计，AI模型选型侧重中文质量 |

### 1.3 当前阶段目标

**本地运行的个人项目（MVP）**，验证核心玩法可行性，为未来可能的商业化积累技术与用户洞察。

#### 主要竞品详细分析

**AI Dungeon**
- 核心功能：开放世界文字冒险，AI生成叙事，多场景类型
- 商业模式：免费层 + 订阅（$14.99-$49.99/月），月收入约$10万
- 优势：品牌认知度最高，叙事自由度极高，跨平台
- 致命缺陷：**完全无TRPG规则系统**；AI频繁失忆、长叙事不连贯；内容审查过严；扫描用户隐私争议
- 对本项目的启示：规则骨架是AI叙事类产品最大的差异化机会

**SillyTavern（开源）**
- 核心功能：连接任意LLM的角色扮演前端，角色卡共享，世界信息Lorebook
- 商业模式：完全免费开源，GitHub Stars 15000+
- 优势：无内容审查，可切换任意模型，社区生态丰富
- 致命缺陷：上手门槛极高（需配API+环境）；**无TRPG规则引擎**；无网格地图；无可视化战斗
- 对本项目的启示：该用户群体（技术型重度用户）是潜在的高质量早期用户

**Character.AI**
- 核心功能：用户创建AI角色，偏向社交陪伴型角色扮演
- 商业模式：免费 + 订阅（$9.99/月），2024年收入$3220万，月活约2000万
- 优势：用户规模和粘性最强（日均使用75分钟），产品易用
- 致命缺陷：**与TRPG完全无关**；内容限制引发多起未成年人安全诉讼；ARPU极低（$0.72）
- 对本项目的启示：易用性设计值得学习，但产品方向完全不同

**Roll20**
- 核心功能：网格地图（动态光照/战争迷雾）、骰子、角色表、实时语音、内容市场
- 商业模式：免费层 + Plus($5.99/月)/Pro订阅 + 内容市场分成
- 用户群：有组队需求的传统TRPG玩家，D&D占60%用户
- 优势：功能最完整的VTT，无需安装，官方D&D授权内容
- 致命缺陷：**完全没有AI功能**，需要真人DM，单人无法游玩；UI老旧；中文支持几乎为零
- 对本项目的启示：网格地图实现参考，但AI是本质差异

**Foundry VTT**
- 核心功能：自托管VTT，200+游戏系统，强大社区插件生态
- 商业模式：DM一次性付费$50，玩家免费
- 优势：最强定制化；一次性付费长期性价比高；插件生态极丰富
- 致命缺陷：需自托管服务器，上手极难；**无AI功能**；中文社区极小
- 对本项目的启示：模组/插件UGC生态是平台护城河，值得在后期规划

**国内竞品**

| 产品 | AI能力 | 核心问题 |
|------|--------|---------|
| 兰亭演绎 | 无 | COC为主，无AI DM，无地图 |
| 骰子云 | 无 | 纯工具属性，用户粘性低 |
| Infinity Planet | 初步（COC） | 早期产品，功能不完善，仅COC |
| 豆包/Kimi/DeepSeek土法跑团 | 有（无规则） | 无规则系统/无记忆/无地图 |

**"土法跑团"（用AI对话框充当DM）在国内大量存在，证明需求真实，但体验极差——这是最直接的可转化用户群。**

## 3. 需求分析

### 3.1 目标用户画像

**用户 A：DnD 轻度爱好者（主要增量市场）**
- 年龄：18-28岁
- 特征：看过DnD直播/视频想尝试，但苦于凑不到团/找不到DM
- 核心需求：低门槛入门、随时能玩、不依赖他人
- 付费意愿：中（$5-15/月）

**用户 B：资深TRPG老玩家（高价值用户）**
- 年龄：22-35岁
- 特征：有跑团经验，希望随时能玩、不依赖固定团队和DM
- 核心需求：规则准确、叙事质量高、存档完善、模组丰富
- 付费意愿：高（$15-30/月）

**用户 C：AI角色扮演爱好者（可转化用户）**
- 年龄：16-25岁
- 特征：C.AI/SillyTavern用户，喜欢AI角色扮演但觉得"不像跑团"
- 核心需求：有规则骨架的沉浸式AI角色扮演体验
- 付费意愿：中低（但基数大）

### 3.2 核心用户故事

```
作为一个DnD爱好者，
我希望能上传一个模组并独自开始冒险，
这样我就不需要等待组团，随时想玩就能玩。

作为玩家，
我希望AI队友有各自的性格和战斗风格，
这样队伍交互感觉像真正的跑团而不是工具调用。

作为玩家，
我希望战斗时能看到网格地图上的位置和血量，
这样战斗过程清晰可跟踪，不只是文字描述。

作为玩家，
我希望能存档并在任意时刻继续上次的冒险，
这样我不需要一次性跑完整个模组。
```

## 4. 产品设计

### 4.1 页面结构

```
/ (首页)
├── 模组列表 + 上传入口
└── 存档列表（继续冒险）

/module/:id/setup (创建角色)
├── 角色创建表单
└── AI队伍预览（生成后展示）

/adventure/:sessionId (跑团主界面)
├── 左侧：队伍状态面板（角色卡 + HP）
├── 中间：跑团日志 + 输入框
└── 右侧：当前场景信息 + 物品栏

/adventure/:sessionId/combat (战斗界面)
├── 左侧：回合顺序 + 角色状态
├── 中间：网格地图（可交互）
└── 右侧：行动选择面板 + 战斗日志
```

### 4.2 跑团主界面设计原则

- **深色主题**，奇幻风格配色（深褐/金色/暗红）
- 日志区域使用不同颜色区分角色（DM叙事/玩家/AI队友/系统提示）
- 检定结果以独立卡片形式弹出（展示骰子结果 + 成功/失败判定）
- 战斗切换平滑过渡（叙事界面 → 战斗地图）

### 4.3 网格战斗地图规格

- 默认 20×20 格，每格代表 1.5米（5英尺）
- 实体用图标+颜色区分（玩家/队友/敌人/NPC）
- 点击格子查看该位置实体信息
- 支持缩放和拖拽
- 每个实体头顶显示简略HP条

### 5.3 数据库设计

```sql
-- 模组表
modules (
  id, name, file_path, file_type,
  parsed_content JSON,     -- 结构化模组数据
  level_min, level_max,
  recommended_party_size,
  created_at
)

-- 角色表（玩家 + AI队友共用）
characters (
  id, session_id,
  is_player BOOLEAN,
  name, race, class, subclass, level, background, alignment,
  ability_scores JSON,     -- {str, dex, con, int, wis, cha}
  derived JSON,            -- {hp_max, ac, initiative, proficiency_bonus...}
  hp_current,
  equipment JSON,
  spells JSON,
  personality TEXT,        -- AI队友专用
  speech_style TEXT,       -- AI队友专用
  combat_preference TEXT   -- AI队友专用
)

-- 游戏会话/存档表
sessions (
  id, module_id, player_character_id,
  game_state JSON,         -- 当前场景、剧情进度flags、已触发事件
  current_scene TEXT,
  session_history TEXT,    -- 压缩后的历史摘要
  combat_active BOOLEAN,
  created_at, updated_at
)

-- 战斗状态表
combat_states (
  id, session_id,
  grid_data JSON,          -- 地形、障碍物
  entity_positions JSON,   -- {entity_id: {x, y}}
  turn_order JSON,         -- 先攻顺序列表
  current_turn_index,
  round_number
)

-- 跑团日志表
game_logs (
  id, session_id,
  role,                    -- dm / player / companion_name / system
  content TEXT,
  log_type,                -- narrative / combat / dice / companion
  dice_result JSON,        -- 若有骰子结果
  created_at
)
```

### 5.4 系统交互流程

```
用户上传模组
    → 后端解析文件文本
    → 调用 Dify WF1（模组解析）
    → 存入 modules 表

用户创建角色 + 确认队伍
    → 调用 Dify WF2（队伍生成）
    → 存入 characters 表
    → 创建 session 记录

用户输入行动
    → 后端接收
    → 调用 Dify WF3（地下城主）
    → 判断是否触发战斗
    ├── 否 → 返回叙事文本 + 队友反应 → 写入 game_logs
    └── 是 → 前端切换战斗界面
              → WebSocket 建立实时连接
              → 逐回合调用 Dify WF4（战斗叙述）
              → 战斗结束后返回探索界面
```

## 6. AI Workflow 设计

### 6.1 Workflow 总览

| 编号 | 名称 | 触发时机 | 输入 | 输出 |
|------|------|---------|------|------|
| WF1 | 模组解析器 | 用户上传模组文件后 | 模组原始文本 | 结构化JSON |
| WF2 | 队伍生成器 | 玩家完成角色创建后 | 玩家职业/等级/模组数据 | AI队友完整配置 |
| WF3 | 地下城主 | 每次玩家输入行动 | 玩家输入/场景/历史/NPC | 叙事文本/队友反应/战斗触发信号 |
| WF4 | 战斗叙述器 | 战斗中每个回合 | 行动者/目标/战场状态 | 骰子结果/战斗叙述/状态更新 |

### 6.2 WF3 地下城主节点逻辑

```
玩家输入
    ↓
[LLM] 分类：combat / dialogue / exploration
    ↓
[If-Else] 是否 combat？
    ├── True → 返回战斗触发信号（前端切换战斗界面）
    └── False
           ↓
        [If-Else] 是否 dialogue？
            ├── True → [LLM] NPC对话处理 → [LLM] 队友反应 → 结束
            └── False → [LLM] 探索叙事处理 → [LLM] 队友反应 → 结束
```

### 6.3 WF4 战斗叙述器节点逻辑

```
战斗回合开始
    ↓
[If-Else] 行动者是否为玩家？
    ├── True（玩家回合）
    │      → [Code] 骰子计算（攻击/技能/法术）
    │      → [LLM] 生成战斗叙述
    │      → 结束
    └── False（AI回合：队友或敌人）
           → [LLM] AI战术决策（选择行动和目标）
           → [Code] 骰子计算
           → [LLM] 生成战斗叙述
           → 结束
```

### 6.4 DSL 文件说明

4个Workflow的完整Dify DSL文件已生成，位于 `dify_workflows/` 目录，可直接在Dify中一键导入：

- `01_module_parser.yml`
- `02_party_generator.yml`
- `03_game_master.yml`
- `04_combat_narrator.yml`

导入后需在每个 LLM 节点手动切换为目标模型（详见第7节）。

## 7. 模型选型

### 7.1 选型原则

1. **叙事节点优先质量**：跑团体验的核心是叙事质量，不在此处省钱
2. **工具性节点优先性价比**：分类、结构化提取等任务用便宜快速的模型
3. **中文优先**：所有叙事类节点选中文质量优秀的模型
4. API 来源：**AiHubMix** 和 **OpenRouter**，模型在 Dify 中可随时切换

## 9. 未来商业化方向

> 本节为概念性规划，仅为未来可能商业化做参考，当前阶段不作为开发目标。

### 9.1 商业化时机判断

当以下条件成立时，可以考虑商业化：
- 核心玩法稳定（Phase 2完成）
- 有稳定的用户留存
- 模组生态初步形成（用户自发上传分享）

### 9.2 潜在商业模式

**模式一：订阅制（参考 AI Dungeon）**
- 免费版：有限次AI调用，单一模组体验
- 标准版（约¥30/月）：无限次跑团，全功能
- 高级版（约¥68/月）：优先排队、更高质量模型、更大模组文件支持

**模式二：模组市场分成**
- 平台提供模组上传与销售基础设施
- 创作者上传付费模组，平台收取20-30%分成
- 参考 Foundry VTT 社区 / DMs Guild 的成功经验

**模式三：API成本转嫁**
- 玩家自填API Key（类SillyTavern模式）
- 平台提供账号管理和API用量统计
- 降低平台边际成本至几乎为零

### 9.3 中国市场特殊考量

- **COC支持**：国内COC用户远多于DnD用户，优先级与DnD相当，增加COC规则引擎可大幅扩大受众
- **内容合规**：商业化前需评估AI生成内容的审核机制
- **支付接入**：微信支付/支付宝接入是标配
- **本地化模组**：与国内TRPG内容创作者合作，引入中文原创模组

### 9.4 技术护城河

| 护城河 | 说明 |
|--------|------|
| 规则引擎 | DnD 5e（未来扩展COC/PF2e）完整实现，竞品无 |
| 模组解析质量 | 持续优化多格式解析准确率 |
| 叙事一致性 | 基于历史摘要的长程记忆机制 |
| 用户模组生态 | UGC模组库是平台核心资产，随时间积累 |
| 中文TRPG知识库 | 用于RAG增强的中文DnD/TRPG知识库 |

### 9.5 风险提示

| 风险 | 概率 | 应对策略 |
|------|------|---------|
| Hasbro/WotC推出官方AI DM | 中 | 专注中文市场+多规则系统，降低对DnD品牌的依赖 |
| AI API成本上升 | 低 | Workflow支持随时换模型，不绑定单一供应商 |
| 竞品快速跟进 | 中 | 加速模组生态建设，用户数据和内容库形成壁垒 |
| 内容生成合规 | 中（国内）| 游戏内容范围明确，实施内容过滤层 |
